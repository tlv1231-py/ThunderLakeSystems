# Project Overview

This is a client-facing portfolio website for my workflow modernization practice. The site exists to win new clients, not to serve as a resume or a generic tech blog.

## Audience

Primary audience:
- Business owners
- Operations leaders
- Finance / analytics leads
- Small business decision makers
- Potential consulting clients

Secondary audience:
- Technically literate prospects who want to verify credibility

## Core Goal

The site should help a visitor quickly understand:
1. What I do
2. Who I help
3. Why I am credible
4. What results I deliver
5. How to contact me

## Positioning

Use this positioning:
- Workflow modernization
- Automation for legacy systems
- Database / spreadsheet / ops process improvement
- Reliable systems built from messy real-world constraints

Avoid generic language like:
- “full-stack developer”
- “tech consultant”
- “IT services”
- “software engineer portfolio”

## Site Structure

Required pages:
- Home
- Case Studies
- Services
- About
- Contact

Optional later:
- GitHub links
- Testimonials
- Articles / insights

## Content Rules

- Lead with outcomes and business value.
- Case studies must be concrete, specific, and client-facing.
- Show problem -> approach -> result.
- Use plain language.
- Keep copy concise and scannable.
- Avoid hype and buzzwords.
- Do not invent client details or measurable results.
- If a number is unknown, leave it out or mark it as a placeholder.

## Design Rules

- Clean, premium, minimal
- Easy to scan
- Strong typography
- Clear CTA on every page
- Mobile-first responsive layout
- Accessible color contrast and semantic HTML

## Conversion Rules

Primary CTA:
- Book a call
- Request an audit
- Contact me

Secondary CTA:
- View case studies

Every page should clearly guide visitors toward contact.

## Technical Rules

- Keep implementation simple and maintainable.
- Use reusable components where appropriate.
- Do not over-engineer.
- Preserve content in separate files when practical.
- Ask before making major structural changes.
- Do not add unnecessary libraries.

## Working Style

- Build one page or feature at a time.
- Prefer small, reviewable changes.
- After implementing a page, summarize what changed and what still needs content.
- If uncertain, ask a clarifying question before proceeding.
