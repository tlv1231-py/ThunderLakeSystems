Field Service Legacy Systems Automation

Replacing Manual Scheduling with Modular Automation in a Field Service Business

A field service business was running its scheduling operation on an MS Access + VBA stack understood by only one person — brittle, undocumented, and dangerous to touch.

MS Access VBA SQL Field Service Operations
01

Context

The Environment

Inherited a fragmented MS Access and VBA stack running the scheduling and operations of an active field service business. Scheduling rules were hardcoded, logic was understood by a single person, and the codebase had grown organically with no documentation, version control, or deployment process.

02

Problem

Brittle by Design

Any modification to the scheduling logic risked breaking production workflows. SQL patterns were incorrect. Hardcoded thresholds were scattered across modules. The wrong VBA module was being used as the execution point. There was no safe path to change anything.

03

What I Did

SuperScheduler

  • Designed and built a modular scheduling engine with interval-driven priority logic — dynamically pulling per-treatment thresholds from a lookup table rather than hardcoding values in code.
  • Built a keyword-matching system that classifies treatment types at runtime using VBA pattern matching, with a structured fallback for unmatched records and an operator-facing report to surface data gaps.
  • Introduced minimum interval enforcement between treatments, preventing premature re-scheduling without any manual checks.
  • Delivered structured version control, object file standards, deployment checklists, and plain-English VBA documentation — giving the system a maintainable foundation it had never had.
  • Diagnosed and corrected root causes including broken SQL join patterns, wrong module usage, and stale field references — without disrupting live production workflows.
04

Result

A System Worth Trusting

The business moved from a fragile legacy setup toward a modular, maintainable scheduling system — with interval-aware prioritization, safer deployment practices, and operator-facing visibility into data quality.

Changes that previously required tribal knowledge can now be made by reading the code. The system documents itself. Production is no longer one edit away from breaking.

05

Why It Matters

Pattern of Work

"Enter a messy legacy environment, diagnose root causes, and deliver a working system incrementally — without breaking production."

This project demonstrates the ability to operate under real constraints — existing architecture, active business operations, zero tolerance for downtime — and still move the system meaningfully forward.

Data Architecture Legacy Systems SQL

Rebuilding Scheduling Rules into a Canonical, Interval-Aware Data Model

A scheduling system's business logic was spread across hardcoded values, inconsistent SQL patterns, and scattered VBA rules — making it nearly impossible to reason about or safely change.

MS Access VBA SQL
01

Context

The Environment

Inherited a scheduling system where treatment logic was spread across hardcoded values, inconsistent SQL patterns, and scattered VBA rules. The underlying data model had grown organically and was difficult to reason about at the rule layer.

02

Problem

Scattered Logic, Platform Constraints

The business logic was hard to maintain because matching behavior, treatment intervals, and update logic were not centralized. MS Access also imposed structural limitations — some changes could not be implemented with standard SQL patterns alone, requiring workarounds that weren't documented and couldn't be repeated safely.

03

What I Did

Canonical Rule Architecture

  • Rebuilt the scheduling logic around a canonical keyword approach, reducing a 234-row hardcoded table to 96 canonical records.
  • Replaced full-string matching with interval-aware keyword classification so treatments could be reasoned about through a cleaner rule model.
  • Handled Access update limitations by pivoting to a VBA recordset loop when correlated subquery updates were not supported — and documented the workaround pattern for reuse.
  • Designed the logic so rule changes could flow through a central data structure instead of being scattered across code.
  • Captured the platform constraints and implementation pattern in plain-English documentation for future maintenance.
04

Result

Simpler Logic, Safer Changes

The scheduling logic became more compact, more consistent, and easier to extend. Matching was simplified into a canonical structure, update behavior was made compatible with Access's limitations, and future rule changes no longer depend on scattered hardcoded values.

05

Why It Matters

Pattern of Work

"Strong database architecture judgment and the ability to solve real platform constraints without forcing a greenfield rebuild."

This project demonstrates how to simplify complex business logic into a maintainable data model — working within real platform constraints, not around them.

Product Build Full-Stack Music Industry

Guest List Management for Music Industry Organizations

A music industry client needed centralized guest list management across multiple shows, bands, and internal teams. The process had been handled informally — no approval flow, no admin visibility, no audit trail.

01

Context

The Environment

A music industry client needed a better way to manage guest lists across multiple shows, bands, and internal teams. The process had been handled informally, with no centralized tracking, approval flow, or admin visibility.

02

Problem

No System, No Visibility

There was no system to collect, review, or approve guest requests at scale. Members submitted requests ad hoc, admins lacked a unified view of capacity, credentials, and status, and there was no audit trail. As show volume grew, the informal process became a liability.

03

What I Did

Dual-Portal Application

  • Designed and built a full-stack guest list management application from scratch — purpose-built for the operational realities of a working music organization.
  • Built a mobile-first member portal for submitting guest requests, with dynamic attendee and subguest management so members can structure requests without back-and-forth.
  • Built a desktop-first admin portal for reviewing requests, managing credentials, tracking per-show capacity, and controlling access — giving admins a single place to manage everything.
  • Implemented role-based access control across three user roles, keeping member, admin, and operator views cleanly separated without complexity for end users.
  • Built in soft deletes, full audit logging, and inline editing throughout the admin view — so all changes are tracked and reversible without technical intervention.
04

Result

Production-Ready from Day One

Delivered a working, production-ready application with seeded real-world data and Vercel-ready deployment. Admins can manage shows end-to-end from a single view. Members can submit structured guest requests in under a minute.

All changes are tracked and reversible. The system handles the operational complexity so the team doesn't have to.

05

Why It Matters

Pattern of Work

"Design and ship a full-stack operational tool fast — with a clean user experience and the controls needed for real-world use."

This project demonstrates product thinking, workflow design, and technical execution beyond legacy system modernization — the ability to identify what an operation actually needs, build it cleanly, and hand it off ready to use.
